Security Report

Web Security Final Project — Memory Game (Frontend + Backend)

← Back to Game Login

Project Summary

This project is a Memory Game that includes a backend API for authentication, comments, and leaderboard. It demonstrates common vulnerabilities and their secure alternatives.

Backend API (used by the frontend)

  • POST /api/register — create a user
  • POST /api/login — login (Express session cookie)
  • POST /api/logout — logout
  • GET /api/me — current session user
  • POST /api/score — save a score
  • GET /api/leaderboard — leaderboard
  • POST /api/comment — add a comment
  • GET /api/comments — list comments

What we demonstrated (Vulnerable)

The following issues represent common real-world web vulnerabilities. Some are shown intentionally (demo), and some can appear when the backend trusts the client too much.

1) Stored XSS (Comments)

  • Where: Comments feature.
  • Exploit: In VULNERABLE mode, comments are rendered using innerHTML (HTML is interpreted instead of displayed as text).
  • Example payload: <img src=x onerror=alert('XSS')>
  • Impact: Attacker can execute JavaScript in the victim’s browser when the stored comment is viewed (session hijacking, UI manipulation, fake actions, etc.).

2) Client-Side Trust (Score Tampering via API)

  • Where: Leaderboard / score submission.
  • Exploit: If the server accepts a score value directly from the client, an attacker can send an arbitrary number (e.g., 999999) or try to submit a score for another user by changing userId in the request body.
  • Impact: Leaderboard integrity is broken (fake scores, unfair ranking).
  • Note: Even if the UI blocks guests from saving, an attacker can still craft requests manually in DevTools/console unless the server enforces authorization.

3) Broken Access Control / IDOR (User Impersonation)

  • Where: POST /api/comment and POST /api/score.
  • Exploit: If the backend uses userId from the request body instead of the session user, an attacker can submit actions as another user by sending a different userId.
  • Impact: Impersonation + fake comments/scores under other accounts.

4) SQL Injection Risk (Authentication)

  • Where: Login/registration database queries.
  • Exploit: If SQL queries are built using string concatenation/interpolation, attackers can inject SQL to bypass authentication or extract data.
  • Impact: Unauthorized access, data leakage, account takeover.
  • Fix: Always use parameterized queries / prepared statements.

Secure version (what we implemented / should implement)

  • Stored XSS fix: render comments as text using textContent (SECURE mode) or sanitize HTML on output.
  • Server-side authorization: protect sensitive routes with authentication middleware (requireAuth) and reject guest actions.
  • Session identity only: ignore userId sent by the client; use req.session.user.id on the server.
  • Score validation: validate score ranges and/or compute score server-side, not trusting the client number.
  • SQL Injection fix: parameterized queries everywhere.
  • Password storage: hash passwords (bcrypt) instead of storing raw passwords.

How to run the demo (steps)

  1. Open index.html → Register → Login.
  2. Open game.html and play a round to create a score.
  3. Go to Comments → switch to VULNERABLE and submit the XSS payload.
  4. Refresh the page and observe that the payload executes (Stored XSS).
  5. Switch to SECURE and submit the same payload (it should display as text, not execute).
  6. (Optional) Open DevTools → Network and inspect /api/score and /api/comment requests to verify session cookies are used.
  7. Use Reset Demo / logout to clean up and repeat.
